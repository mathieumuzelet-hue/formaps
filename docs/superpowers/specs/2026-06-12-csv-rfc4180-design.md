# Spec — PR ③ CSV RFC 4180 (exports durcis + import robuste)

**Date** : 2026-06-12
**Origine** : audit complet 2026-06-09 (PR ③ de la roadmap consolidée : A1 injection de
formule, D2 cap bulk, + inédits Fable : `__parsed_extra`, Windows-1252, newlines
faq-gaps).

## Objectif

Rendre les exports CSV destinés aux humains sûrs (injection de formule Excel) et
conformes RFC 4180 (quoting), et l'import CSV robuste (encodage Excel FR, colonne en
trop, cap de taille). L'export FAQ Builder → Dify n'est PAS modifié (ingestion
machine, décision PR #12).

## 1. Helper partagé `csvCell()` — nouveau `src/lib/csv.ts`

- `csvCell(value: string): string` :
  1. **Guard anti-formule d'abord** : si le 1er caractère est `=`, `+`, `-`, `@`,
     TAB (`\t`) ou CR (`\r`) → préfixe `'` (neutralise l'interprétation formule
     d'Excel sur du contenu cross-user, cible = l'admin qui ouvre l'export).
  2. **Quoting RFC 4180 ensuite** : si la valeur (éventuellement préfixée) contient
     `;` (délimiteur du projet), `"`, `\n` ou `\r` → doubler les `"` internes et
     entourer de `"`.
  - Une cellule saine ressort inchangée. Module browser-safe, zéro dépendance.
- `DELIMITER` (`;`) et `BOM` migrent dans `src/lib/csv.ts` (réexportés par
  `csv-export.ts` pour ne pas casser les imports existants, ou imports mis à jour —
  au choix de l'implémentation, mais une seule source de vérité).

## 2. Application aux exports humains

**Fichiers** : `src/lib/admin/csv-export.ts`, `src/lib/admin/faq-gaps.ts`.

- `toCredentialsCsv` : `csvCell(email)` + `csvCell(password)` (le charset des mots de
  passe générés exclut `=+-@`, mais l'email reste vecteur — et le helper est
  systématique).
- `buildTemplateCsv` : cellules passées par `csvCell` (contenu statique sain — c'est
  l'uniformité du contrat qui compte).
- `buildFaqGapsCsv` : le remplacement `;`→`,` est SUPPRIMÉ ; chaque cellule passe par
  `csvCell`. Les questions BRAIN multi-lignes ou contenant `;`/`"` deviennent des
  cellules quotées valides (fidélité restaurée), les questions commençant par
  `=+-@` sont neutralisées (A1 — vecteur cross-user principal).
- **Non modifié** : `src/lib/faq/csv.ts` (export Dify mode Q&A) — un test de
  non-régression verrouille que sa sortie reste byte-identique.

## 3. Import : encodage Windows-1252

**Fichiers** : nouveau `src/lib/admin/decode-csv-file.ts` (ou intégré à un module
existant browser-safe), `src/components/admin/CsvImportCard.tsx`.

- `decodeCsvBytes(bytes: Uint8Array): string` (pur, testable) :
  - tente `new TextDecoder('utf-8', { fatal: true }).decode(bytes)` ;
  - si ça lève (octets 1252 invalides en UTF-8, ex. `é` = 0xE9) → retour
    `new TextDecoder('windows-1252').decode(bytes)` ;
  - le BOM UTF-8 éventuel est retiré (TextDecoder utf-8 le strip nativement ;
    vérifier le chemin 1252).
- `decodeCsvFile(file: File): Promise<string>` = wrapper `arrayBuffer()` →
  `decodeCsvBytes`.
- `CsvImportCard` : les deux appels `Papa.parse<…>(file, …)` reçoivent désormais la
  **string décodée** (papaparse accepte une string en entrée ; l'option de config
  reste identique sinon).

## 4. Import : strip `__parsed_extra`

**Fichier** : `src/components/admin/CsvImportCard.tsx` (post-parse, avant tout envoi
et avant `parseStoreRows`/`parseUserRows`).

- Pour chaque ligne parsée : supprimer la clé `__parsed_extra` (papaparse l'ajoute,
  valeur `string[]`, dès qu'une ligne a une colonne de plus que le header → le
  `z.record(z.string(), z.string())` serveur rejette TOUT l'import — vérifié par
  exécution à l'audit). Une ligne avec un `;` superflu redevient importable, ses
  colonnes connues faisant foi.
- Le schéma serveur `bulkImportSchema` reste strict (defense in depth, pas de
  tolérance côté serveur).

## 5. Import : cap 2000 → 200

**Fichiers** : `src/server/trpc/routers/admin.ts` (`bulkImportSchema`),
`src/components/admin/CsvImportCard.tsx`.

- `bulkImportSchema` : `.max(2000)` → `.max(200)` (~200 hash argon2 × ~200 ms restent
  sous les timeouts proxy ; D2).
- Client : si le fichier parsé dépasse 200 lignes de données → message d'erreur
  explicite AVANT tout appel réseau (« Le fichier dépasse 200 lignes — le scinder en
  plusieurs imports. »), pas d'erreur zod brute.

## Hors scope (assumé)

Import transactionnel / reprenable (le skip unique-violation existant rend le
ré-import possible) ; export Dify FAQ Builder ; détection d'encodages exotiques
(UTF-16…) ; PR ⑦ embed-lab.

## Tests (TDD)

- `tests/lib/csv.test.ts` : cellule saine inchangée ; quoting sur `;`, `"`, `\n`,
  `\r` (doubles quotes doublées) ; guard sur chacun de `= + - @ \t \r` en tête ;
  guard PUIS quoting combinés (`=a;b` → `"'=a;b"`) ; chaîne vide.
- `csv-export` : credentials avec email `=evil@x.fr` neutralisé ; template inchangé
  hors uniformisation ; snapshot de forme (BOM + `;`).
- `faq-gaps` : question multi-lignes → une seule ligne logique quotée ; question avec
  `;` conservée telle quelle (plus de virgule de substitution) ; question `=...`
  neutralisée.
- Non-régression : sortie de `src/lib/faq/csv.ts` byte-identique sur un échantillon.
- `decode-csv-file` : bytes UTF-8 purs → identité ; bytes 1252 (`é`=0xE9, `€`=0x80)
  → décodés correctement ; UTF-8 avec BOM → BOM absent du résultat.
- `csv-import`/CsvImportCard : ligne avec `__parsed_extra` strippée puis validée ;
  >200 lignes → message client, aucun appel mutation (test composant) ; serveur :
  201 lignes → rejet zod.

## Déploiement

Aucun env, aucune migration. Changement visible : les exports credentials/faq-gaps
peuvent contenir des cellules quotées (`"..."`) — Excel les ouvre nativement.
